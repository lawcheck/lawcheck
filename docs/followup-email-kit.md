# Письмо-догонялка для лидов (scan_submit → оплата)

_Составлено 2026-07-17. Цель: дожимать узкое место воронки — люди запустили
бесплатную проверку, оставили email на отчёте (`Lead`), но не оплатили Pro.
Эквайринг Точки подключён (2026-06-22), оплата с отчёта работает — то есть
догонялка ведёт не в тупик, а на живую кассу._

## 0. Кого догоняем (сегмент)

Лид подходит под рассылку, если:
- есть запись в `leads` (email оставлен на странице отчёта);
- по его `scan_id` (и по email) **нет** оплаченного `Order` (`status='paid'`);
- у скана есть находки с `severity != 'ok'` (есть чем пугать/помогать);
- прошло **24–72 ч** с `Lead.created_at` (не в лоб, дать «остыть»);
- письмо этому email ещё не отправляли (нужен флаг — см. §4).

## 1. Чем персонализируем (всё уже в БД)

Из `Lead`: `url`, `scan_id`, `created_at`.
Из `Finding` по `scan_id`: `title`, `severity`, `law_reference`, `location`,
число проблем и число скрытых рецептов (`locked_count` = проблемы − `_FREE_RECIPES`).

Плейсхолдеры письма:
`{site}` — Lead.url без схемы · `{problems}` — число находок severity!=ok ·
`{critical}` — число critical · `{top3}` — 3 заголовка худших находок ·
`{laws}` — уникальные `law_reference` (напр. «ст. 18.1, 18 152-ФЗ») ·
`{locked}` — сколько готовых текстов «Как исправить» скрыто под Pro ·
`{report_url}` — https://lawchek.ru/report/{scan_id} · `{unsub_url}`.

## 2. Тема письма (A/B, без капса и спам-слов)

- A (риск): `{site}: {critical} критичных нарушения по 152-ФЗ — что грозит`
- B (польза): `Ваш отчёт по {site} готов: {problems} находок и как их закрыть`
- C (мягкая): `Не закончили с проверкой {site}? Вот что мы нашли`

Рекомендация: старт с B (транзакционная рамка «ваш отчёт» — безопаснее по
рекламе, выше open rate), A держать на второй волне.

## 3. Тело письма (HTML-friendly, текст ниже плейсхолдеров)

> Здравствуйте!
>
> Вы запускали проверку сайта **{site}** на LawCheck. Собрали для вас итог,
> чтобы не потерялся.

> Самое важное:
> - {top3[0]}
> - {top3[1]}
> - {top3[2]}
>
> Это зона {laws} — по ней Роскомнадзор штрафует бизнес: с 30 мая 2025 г.
> отсутствие/ошибки в политике и согласиях — до 300 000–500 000 ₽ для юрлиц,
> утечка ПДн — до 15 млн ₽. Проверка сама по себе штраф не убирает —
> нарушения надо закрыть.
>
> В бесплатном отчёте открыты первые рекомендации. Ещё **{locked}** готовых
> текстов «Как исправить» (политика ПДн, тексты согласий, cookie-баннер под
> ваш сайт) — в Pro за 990 ₽/мес: генерируем документы под {site},
> показываем, что и куда вставить, и следим за сайтом, чтобы не откатилось.

> → Забрать готовые тексты (Pro): {report_url}#pro

> Вы получили письмо, потому что оставили email для отчёта по {site}.

Тон: помощь, не давление. Один явный CTA (открыть отчёт), Pro — вторым.
Дисклеймер «носит рекомендательный характер, не юруслуга» — как на сайте.

## 4. Юридическая рамка (важно — это профиль самого продукта)

- **ФЗ «О рекламе» ст. 18**: реклама по email — только с предварительного
  согласия. Лид оставил адрес «прислать отчёт + следить за сайтом» → письмо
  **про его отчёт** = сервисное/ожидаемое; Pro-блок делает его пограничным.
  Держим транзакционную рамку в теме и первом экране, Pro — ниже.
- **Отписка обязательна**: сейчас механизма нет. Нужен `unsub_token` у Lead и
  `GET /unsubscribe/{token}` → помечает `Lead.unsubscribed_at`. Без этого не
  отправлять.
- **152-ФЗ**: email — это ПДн; в Политике ПДн упомянуть рассылку сервисных
  писем и уведомлений по оставленному адресу (свериться с текущей privacy.html).
- Первую волну (до автоматизации) слать вручную по 5–10 адресов из
  `/inbox`, руками смотреть отчёт — так отловим тон и возражения.

## 5. Реализация — СДЕЛАНО (2026-07-17)

- **Миграция**: `Lead` + `unsub_token`, `mailed_at`, `unsubscribed_at`;
  лёгкая идемпотентная миграция в `db/session.py::_migrate_leads_followup`
  (create_all не меняет существующую таблицу) + бэкфилл токенов старым лидам.
- **Отбор**: `repo.leads_to_followup(delay_hours=24, max_age_days=14, limit)` —
  не писали/не отписались/возраст в окне, скан done с нарушениями, по
  scan_id и email нет оплаченного Order.
- **Письмо**: `reporting/followup.py` — `build_context` (плейсхолдеры из
  Scan+Findings, включая суммарный риск штрафа) + `render` (HTML+text, тема
  «польза», русское склонение по числам) + `send_one`/`run`.
- **Отписка**: `GET /unsubscribe/{token}` (отписывает все лиды с этим email) +
  ссылка в футере каждого письма.
- **Запуск батча**: `python -m lawcheck.tools.send_followups [--dry-run]
  [--limit N]` — в контейнере web (ручной прогон), либо cron-эндпойнт (§6).
  CTA ведёт на отчёт, с него — живая оплата (эквайринг подключён).
- **UTM**: ссылка на отчёт помечена
  `utm_source=email&utm_medium=email&utm_campaign=followup` — в Метрике видно
  воронку письмо → отчёт → оплата отдельно от рекламы. Ссылка отписки —
  служебная, без меток.
- **Тесты**: `tests/test_followup.py` (28) — отбор, контекст, UTM, склонение,
  отправка/dry-run/фейл, отписка, cron-эндпойнт (в т.ч. 403 без ключа).

## 6. Расписание (cron)

По конвенции проекта периодика = внутренний эндпойнт + хостовый cron
(как `/internal/monitoring/run`). Эндпойнт: `POST /internal/followups/run`,
заголовок `X-Internal-Key: $INTERNAL_KEY`, query `?limit=20&dry_run=false`.

Скрипт на сервере `/root/bin/lawcheck-followups.sh` (по образцу
`lawcheck-monitoring.sh`; в git не хранится — как и остальные cron-скрипты):

```sh
#!/bin/sh
# Письма-догонялки лидам: дергаем внутренний эндпойнт API раз в сутки.
KEY=$(grep "^INTERNAL_KEY=" /root/lawcheck/.env | cut -d= -f2)
curl -fsS -X POST -H "X-Internal-Key: $KEY" \
  "https://lawchek.ru/internal/followups/run?limit=20"
echo " $(date -Is)"
```

```
chmod +x /root/bin/lawcheck-followups.sh
# crontab -e — 11:00 МСК ежедневно (рабочее время B2B):
0 11 * * * /root/bin/lawcheck-followups.sh >> /var/log/lawcheck-followups.log 2>&1
```

`limit=20/сутки` — осознанно мало: бережём репутацию домена, льём постепенно.
**Ставить cron только ПОСЛЕ деплоя этого кода** (иначе 404).

## 7. SMTP и репутация домена — проверено 2026-07-17

Прод-конфиг (`/root/lawcheck/.env`) и DNS сходятся, база для доставляемости
здоровая:

| Что | Факт | Вывод |
|---|---|---|
| SMTP_HOST | `smtp.timeweb.ru` | покрыт SPF ✓ |
| SMTP_PORT / STARTTLS | `465` / `false` | верно, mailer идёт в SMTP_SSL ✓ |
| SMTP_FROM | `LawCheck <noreply@lawchek.ru>` | домен совпадает → DMARC-выравнивание ✓ |
| SPF | `v=spf1 include:_spf.timeweb.ru ~all` | отправка через Timeweb авторизована ✓ |
| DKIM | селектор `dkim`, RSA-ключ есть | подпись Timeweb ✓ |
| DMARC | `v=DMARC1; p=quarantine;` | политика строгая ✓ |

**Единственный пробел — в DMARC нет `rua=`**: агрегированные отчёты никуда не
шлются, доставляемость мы не видим. Перед первой массовой волной добавить TXT
`_dmarc.lawchek.ru`:
`v=DMARC1; p=quarantine; rua=mailto:postmaster@lawchek.ru; fo=1;`

Прочее перед волной:
- Первую волну слать вручную (`--limit 5`), смотреть отказы в логах, только
  потом включать cron.
- `handoff/ACCESS.md` устарел: в списке прод-`.env` не перечислены `SMTP_*`
  (фактически заданы) — дописать при следующей ревизии.

## 8. Открытые вопросы

  ходят, значит да; подтвердить `smtp_from` и репутацию домена (SPF/DKIM),
  иначе догонялки уйдут в спам.
- Частота: одно письмо на лид или серия из 2 (день 1 «отчёт», день 4 «риск»)?
  Начать с одного, серию — после первых цифр.
